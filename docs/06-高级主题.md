# 06-高级主题

本页只讨论仓库当前真实具备的高级能力，以及它们在 `examples/` / `test/` / `benchmark/` 中的验证状态。

## 高级能力矩阵

| 主题 | 真实 API / 资产 | 当前验证状态 | 说明 |
|------|-----------------|--------------|------|
| SNI / Hostname | `SslEngine::setHostname()`、`SslSocket::setHostname()` | 已验证 | `examples/include/E2-ssl_client.cc` 与 `test/T2-ssl_loopback_smoke.cc` 都会设置主机名 |
| CA 校验 | `loadCACertificate()`、`setVerifyMode(Peer)` | 已验证 | `E2-SslClient-Include` 传 `certs/ca.crt` 可校验；`T2` 为更强验证链路 |
| ALPN 协商 | `SslContext::setALPNProtocols()`、`SslEngine::getALPNProtocol()`、`SslSocket::getALPNProtocol()` | 已验证 | `T3-SslSingleShotSemantics` 在双端配置 `{"h2","http/1.1"}`，并断言协商结果为 `h2` |
| Session 复用 | `setSessionCacheMode()`、`setSessionTimeout()`、`setSession()`、`getSession()`、`isSessionReused()` | 已验证 | `T3-SslSingleShotSemantics` 复用第一次连接的 `SSL_SESSION`，第二次连接断言 `isSessionReused()==true` |
| CA 路径 | `loadCAPath()` | 已验证 | `T1-SslSocketTest` 会创建带 OpenSSL hash 文件名的临时 CA 目录并验证加载成功 |
| 系统默认 CA | `useDefaultCA()` | 环境依赖 API 级 | 默认信任库取决于运行环境，仓库当前没有独立跨平台 smoke target |
| TLS 版本约束 | `setMinProtocolVersion()`、`setMaxProtocolVersion()` | 已验证 | `T1` 校验 `SSL_CTX` 原生 getter；`T3` 强制双端使用 TLS 1.2 |
| Cipher 策略 | `setCiphers()` | 已验证 | `T1` 覆盖 setter 回归；`T3` 在 TLS 1.2 loopback 中实际使用约束后的策略 |
| TLS 1.3 CipherSuite | `setCiphersuites()` | API 回归 | `T1` 仅验证 setter 调用成功；仓库还没有独立 TLS 1.3 E2E target |
| mTLS | `loadCertificate()`、`loadPrivateKey()`、`setVerifyMode(FailIfNoPeerCert)`；`test/certs/client.crt` / `client.key` | 资产存在，未落地 E2E | 仓库有客户端证书，但没有现成 target 走双向认证 |
| C++23 Modules | `galay-ssl/module/galay.ssl.cppm`、`galay-ssl-modules` | 条件生成 | 安装包不导出模块 target |

## ALPN

真实来源：

- `galay-ssl/ssl/SslContext.h`
- `galay-ssl/ssl/SslEngine.h`
- `galay-ssl/async/SslSocket.h`
- `test/T3-ssl_single_shot_semantics.cc`

当前仓库能确认的事实：

- `SslContext::setALPNProtocols()` 现在会在客户端配置 advertised list，在服务端配置选择回调
- `T3-SslSingleShotSemantics` 会在 client/server 两侧同时设置 `{"h2","http/1.1"}`
- `T3` 对 `SslSocket::getALPNProtocol()` 与服务端 socket 的协商结果都断言为 `h2`

因此结论是：ALPN 已不只是“API 存在”，而是已有真实自动化 smoke target。

## Session 复用

真实来源：

- `SslContext::setSessionCacheMode()`
- `SslContext::setSessionTimeout()`
- `SslEngine::setSession()` / `getSession()` / `isSessionReused()`
- `SslSocket` 对应包装方法
- `test/T3-ssl_single_shot_semantics.cc`

当前仓库能确认的事实：

- `T3-SslSingleShotSemantics` 第一次连接会缓存 `SSL_SESSION`
- 第二次连接会在握手前调用 `SslSocket::setSession()`
- 第二次连接握手后，客户端与服务端都断言 `isSessionReused()==true`

因此文档可以把 Session 复用描述为“已有测试闭环的公开能力”，但不能把它扩展成“已有 benchmark 结论”。

## CA 路径与系统 CA

真实来源：

- `SslContext::loadCAPath()`
- `SslContext::useDefaultCA()`
- `test/T1-ssl_socket_test.cc`

当前仓库能确认的事实：

- `T1-SslSocketTest` 会构造一个符合 OpenSSL 目录命名规则的临时 CA 目录，验证 `loadCAPath()` 能真实成功
- `useDefaultCA()` 是公开 API，但它依赖运行环境的默认信任库布局
- 因为不同平台的默认 trust store 差异较大，仓库当前没有把 `useDefaultCA()` 写成“跨平台必过”的 smoke test

因此结论是：`loadCAPath()` 已有自动验证；`useDefaultCA()` 仍应视为环境依赖能力。

## TLS 版本与 Cipher 策略

真实来源：

- `SslContext::setMinProtocolVersion()`
- `SslContext::setMaxProtocolVersion()`
- `SslContext::setCiphers()`
- `SslContext::setCiphersuites()`
- `test/T1-ssl_socket_test.cc`
- `test/T3-ssl_single_shot_semantics.cc`

当前仓库能确认的事实：

- `T1` 会直接检查 `SSL_CTX_get_min_proto_version()` / `SSL_CTX_get_max_proto_version()` / `SSL_CTX_get_session_cache_mode()` / `SSL_CTX_get_timeout()`
- `T3` 强制 client/server 都走 TLS 1.2，并在受限 Cipher 策略下完成两次 loopback 连接
- `setCiphersuites()` 目前只有 setter 级回归，没有独立 TLS 1.3 E2E target

因此结论是：TLS 版本约束与 TLS 1.2 Cipher 策略已具备真实测试闭环；TLS 1.3 CipherSuite 仍是 API 回归级覆盖。

## mTLS

仓库当前的事实边界：

- `test/certs/` 内已有 `client.crt` 与 `client.key`
- `SslVerifyMode` 里也有 `FailIfNoPeerCert`
- 但没有现成 target 把这些资产组合成双向认证的可执行示例

因此本仓对 mTLS 的状态是：

- 证书资产：有
- API 基础：有
- 已验证端到端示例：无

## 超时策略

需要特别说明：

- `SslErrorCode` 中存在 `kHandshakeTimeout` 与 `kTimeout`
- 但当前公开头中没有 `SslSocket::setTimeout()` 之类的显式超时配置接口

这意味着：

- 文档可以说“库会暴露 timeout 类错误码”
- 不能说“仓库已经提供某个公开 timeout setter 或完整超时示例”

## C++23 Modules

仓库实际行为：

- 根工程只在 `BUILD_MODULE_EXAMPLES=ON`、CMake `>= 3.28`、生成器支持模块且编译器侧模块依赖扫描可用时创建 `galay-ssl-modules`
- 若扫描能力缺失（例如 macOS `AppleClang 17` + CMake `4.0.2` + `Ninja` 的常见命令行工具链组合），根工程会 warning 并自动关闭 `BUILD_MODULE_EXAMPLES`
- `examples/import/*.cc` 依赖该 target
- 已安装包不会导出模块 target

因此推荐策略是：

- 源码树实验模块：可以
- 作为安装后稳定消费入口：当前不建议依赖 modules target

## 仍需你自行组合的高级能力

下面这些需求，当前更适合你基于 API 自行组合，而不是直接寻找仓库内现成 target：

- 双向证书认证（mTLS）
- `useDefaultCA()` 在不同平台上的部署策略
- TLS 1.3 CipherSuite 的端到端回归
- 依赖调度器策略的超时治理

如果你需要仓库内现成可运行链路，请优先使用：

- `examples/include/E1-ssl_echo_server.cc`
- `examples/include/E2-ssl_client.cc`
- `test/T2-ssl_loopback_smoke.cc`
- `test/T3-ssl_single_shot_semantics.cc`
