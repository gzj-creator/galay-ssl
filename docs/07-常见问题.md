# 07-常见问题

## 为什么文档顺序从 `00` 开始？

因为仓库里真实存在的入口页就是 `docs/00-快速开始.md`。本次修复没有重命名源码旁文档，而是把：

- 文件名编号
- 页面标题编号
- README / `docs/README.md` 导航顺序

统一到同一套 `00` 到 `07` 编号上。

## `./scripts/run.sh test` 为什么没有跑 `T2-SslLoopbackSmoke`？

因为旧文档曾经只写了 `T1`，但当前脚本 `scripts/run.sh` 的 `test` 分支会顺序执行：

```bash
ctest --test-dir build --output-on-failure
```

如果你想手动重跑端到端 loopback 验证，也可以直接运行：

```bash
./build/bin/T2-SslLoopbackSmoke
```

## `./scripts/check.sh` 会做完整验证吗？

不会。它只会：

- 重新运行 `ctest --test-dir build --output-on-failure`
- 检查 benchmark 二进制是否存在
- 检查证书是否复制到 `build/bin/certs`

它不会：

- 启动真实 benchmark 测试
- 验证 include/import 示例

## 为什么 import 示例有时根本没生成？

因为 `examples/import/*.cc` 依赖 `galay-ssl-modules`，而该 target 只有在下列条件同时成立时才会创建：

- `BUILD_MODULE_EXAMPLES=ON`（默认值是 `OFF`）
- CMake `>= 3.28`
- 生成器是 `Ninja` 或 `Visual Studio`
- 编译器侧模块依赖扫描可用；否则根工程会 warning 并自动关闭该选项

已实测：macOS `AppleClang 17` + CMake `4.0.2` + `Ninja` 在扫描能力缺失时不会生成这些 target。

不满足时，根工程会跳过 import 示例 target。

## 安装后能直接 `import galay.ssl;` 吗？

当前不应这样假设。

真实情况是：

- 仓库内有 `galay-ssl/module/galay.ssl.cppm`
- 但已安装包不导出 `galay-ssl::galay-ssl-modules`

因此安装后稳定消费入口应视为：

- `find_package(galay-ssl REQUIRED)`
- `target_link_libraries(... PRIVATE galay-ssl::galay-ssl)`
- 头文件 include 方式

## 示例和 benchmark 应该从哪里运行？

建议从仓库根目录运行。

原因：

- include 示例和手动 benchmark 命令默认引用根目录的 `certs/`
- 脚本内部如果需要 `build/bin/certs`，会自己切换目录并依赖构建复制动作

## 仓库现在已经验证了哪些高级能力？

已验证：

- 基础 TLS Echo
- CA 证书校验
- SNI/hostname 设置
- ALPN 协商
- Session 复用
- TLS 版本约束与 Cipher 策略
- CA 路径加载
- 基础 benchmark target 可构建

尚未有独立 E2E target 的高级能力：

- mTLS
- 系统默认 CA 的跨平台等价性
- 显式超时配置

## 为什么文档里不再直接给出“最新性能数字”？

因为没有新的原始 benchmark 输出时，这类数字会迅速过时。

本次修复后的规则是：

- 没有新命令输出支撑，就只写 benchmark 方法和 target
- 若要发布数字，必须附上真实命令、环境和原始结果
